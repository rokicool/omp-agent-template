# .DEVREADME.md — Developer & Maintainer Guide

Everything **not** for end users: how the two-plugin set is structured, how
enforcement works internally, how to extend it (agents, skills, extensions), how
to build/test/validate, how to cut a release, and troubleshooting. For the
user-facing install/usage guide, see [README.md](./README.md).

## What this repo is

`omp-agent-template` publishes **two oh-my-pi plugins from one tree**:

| Plugin | Mechanism | Provides |
|---|---|---|
| **`omp-agent-gate`** (A) | extension-package (`package.json#omp.extensions`) | `tool_call` enforcement gate, Definition-of-Done rule, dot-agreement, mess-transport & subagent-panel extensions |
| **`orchestrator-agents`** (B) | marketplace (`.omp-plugin/marketplace.json`) | 7 agent definitions + 8 skills |

oh-my-pi has two disjoint plugin mechanisms — **extension-package** (the only way
to load TypeScript extensions + rules) and **marketplace** (the only way to ship
agent definitions) — so the repo publishes both from one tree.

Current version: **1.8.0**, kept in lockstep across `package.json#version` and
`.omp-plugin/marketplace.json` (`metadata.version` and `plugins[].version`).

## Repo layout

```

.github/workflows/release.yml     # tag -> artifacts + GitHub Release

```

## Architecture: how enforcement works

Two hard layers (runtime blocks the model cannot bypass) plus advisory layers
that reinforce them.

**Hard layers**

1. **Root gate** — `src/enforce-orchestrator.ts` (Plugin A). A `tool_call`
   handler that blocks every tool outside Elon's contract for the interactive
   root session (`ctx.hasUI === true`, project opted in):
   - `read` / `ask` / `todo` → allowed
   - `task` → allowed only when `agent` ∈ `TEAM` (`reqguru, drpe, leaddev,
     validator, docworm, hr`)
   - `write` → allowed only for `.app/PROJECT.md`
   - `bash` → allowed only for `git …`
   - everything else (`edit`, `ast_edit`, `debug`, `browser`, `eval`,
     `web_search`, `find`, `search`, `lsp`, `irc`, …) → blocked
   - Subagents (`ctx.hasUI === false`) are never gated here — they are restricted
     by their own frontmatter.

2. **Agent frontmatter** — `plugins/agents/agents/*.md` (Plugin B). Each agent's
   `tools:` / `spawns:` is enforced at the harness level. A subagent physically
   cannot call a tool not in its list, or spawn an agent not in its `spawns`.

| Agent | `tools` | `spawns` | Role |
|---|---|---|---|
| **Elon** (root) | read, ask, todo, write(.app/PROJECT.md), bash(git), task | reqguru, drpe, leaddev, validator, docworm, hr | Orchestrator — never implements |
| **ReqGuru** | read, write, search, find, mess-send, mess-fail | — | Requirements / grill interview |
| **DrPe** | web_search, read, browser, edit, write, mess-send, mess-fail | — | Research |
| **LeadDev** | read, write, edit, bash, search, find, ast_grep, ast_edit, lsp, debug, task, mess-send, mess-fail | middev, hr | Spec / review / integration; delegates to MidDev |
| **MidDev** | read, write, edit, bash, search, find, ast_grep, ast_edit, lsp, debug, mess-send, mess-fail | — | Implementation |
| **Validator** | read, search, find, lsp, bash, mess-send, mess-fail | — | Spec-vs-impl audit (read-only) |
| **DocWorm** | read, write, edit, search, find, mess-send, mess-fail | — | Documentation |
| **HR** | read, write, edit, mess-send, mess-fail | — | Agent definition / hiring |

`tools`/`spawns` are enforced by oh-my-pi (agent frontmatter for subagents; the
gate for the root) — not advisory. Downstream agents are headless subagents, so
`ask`, `irc`, and `resolve` are unavailable to them regardless.

**Advisory layers** (prompt-level; reinforce the hard invariants):
`APPEND_SYSTEM.md` (Elon framing), `RULES.md` (sticky invariant), `AGENTS.md`,
`PROTO.md`. These ship in `scaffold/` for optional project-local copy; the live
orchestrator protocol ships via `skill://elon` in Plugin B.

## Opt-in mechanism

The gate registers in every project that installs Plugin A, but the `tool_call`
handler early-returns unless the project opts in. Precedence —
`optedIn()` in `src/enforce-orchestrator.ts`:

1. `OMP_BYPASS_ORCHESTRATOR=1` → registers nothing (full escape hatch)
2. `OMP_ENABLE_ORCHESTRATOR=1` → ON
3. `<cwd>/.omp/elon.json` `{"enabled": true}` → ON
4. otherwise → dormant (a malformed or absent marker is dormant — fail-safe)

## APPEND_SYSTEM framing (and why it's advisory)

`APPEND_SYSTEM.md` (the Elon role framing) is **not plugin-shippable** by either
mechanism, so Plugin A's extension **re-injects** it once per `session_start` as
an advisory custom message (`display:false`, queued via `sendMessage` for the
next turn). A project-local `<cwd>/.omp/APPEND_SYSTEM.md` overrides the bundled
default (`src/append-system.default.md`).

This is advisory only: no `ExtensionAPI` call yields a true system-attributed
block (`MessageAttribution` is `"user" | "agent"`; the `Message` union has no
`SystemMessage`; `getSystemPrompt()` is read-only; `appendEntry` is not sent to
the LLM). Do not rely on the prompt alone — the gate enforces the contract. The
framing points Elon at `skill://elon` for the full protocol and does **not**
require `AGENTS.md`/`PROTO.md` to exist in the project.

## Scaffold docs

`scaffold/{AGENTS,PROTO,APPEND_SYSTEM,RULES}.md` are **advisory**
project-local reference/override docs. The orchestrator does **not** need them —
the live protocol ships via `skill://elon` (Plugin B) and the framing via
Plugin A's re-inject, so the one-liner installer needs no extra copy step. To
drop local copies/overrides into a project that wants them:

```bash
cp -r omp-agent-template/scaffold/. .
# → AGENTS.md, PROTO.md, APPEND_SYSTEM.md, RULES.md
```

- `AGENTS.md` — agent registry, enforcement layers, error/recovery, concurrency.
- `PROTO.md` — the full development protocol (phases, artifacts, gates, commit conventions).
- `APPEND_SYSTEM.md` — local override of the Elon framing.
- `RULES.md` — sticky invariant (re-attached every turn, survives compaction).

## Extending: add or change an agent

Agents live at `plugins/agents/agents/<name>.md` with enforced frontmatter:

```yaml
---
name: leaddev
description: Lead developer. Expert software engineer across the full stack. Designs, architects, reviews, and integrates code.
tools: read, write, edit, bash, search, find, ast_grep, ast_edit, lsp, debug, task
spawns: middev, hr
---

# LeadDev — Lead Developer / Architect
```

The `tools:`/`spawns:` lists are the hard contract; the body is advisory. To add
an agent: drop a new `<name>.md`, then **register it** so both Plugin B and the
gate know about it:

1. Add the name to `.omp-plugin/marketplace.json` → `plugins[].agents` (Plugin B
   catalog).
2. If Elon (the root) should spawn it directly, add it to the `TEAM` array in
   `src/enforce-orchestrator.ts`. Otherwise it can only be spawned by an agent
   that lists it in `spawns:` (e.g. `middev` is reached only via `leaddev`).

The agent's behavioral protocol lives in its **skill**, co-located at
`plugins/agents/skills/<name>/SKILL.md` and injected via
`context="skill://<name>"`. The agent definition enforces the **tool boundary**;
the skill defines the **procedure**.

> **HR agent** automates this: for a full hire it produces an agent definition
> and a skill file, and conditionally appends a row to a project-local
> `AGENTS.md` (never installed plugin files). See `skill://hr`.

## Extending: add or change a skill

Skills live at `plugins/agents/skills/<name>/SKILL.md`. The roster ships 8:
`elon`, `hr`, `middev`, `reqguru`, `validator`, `drpe`, `leaddev`, `docworm`. A
skill is plain markdown loaded by omp's skills provider from the plugin's
`skills/` tree and resolved as `skill://<name>`. No registration beyond placing
the directory is needed for resolution, but `scripts/validate-plugins.sh` checks
coverage — keep skills and agents consistent.

## Extending: add an extension

Plugin A extensions are listed in `package.json`:

```json
"omp": { "extensions": ["./src/enforce-orchestrator.ts", "./src/mess-transport.ts", "./src/dot-agreement.ts", "./src/subagent-panel.ts"] }
```

Each is a TypeScript module exporting a default
`(pi: ExtensionAPI) => void | Promise<void>`. Add the file, add it to the array,
and it loads wherever Plugin A is installed. `package.json#files`
(`["src", "rules"]`) controls what `npm pack` ships — keep it in sync when you
add files the extension reads at runtime (e.g. `src/append-system.default.md`).

## The orchestrator protocol

The phase workflow (REQUEST → GRILL → [RESEARCH] → SPEC → DEVELOP ⇄ VALIDATE →
DONE, plus the TRIVIAL short path), artifact ownership in `.app/`, commit
conventions, and the 3-cycle DEVELOP ⇄ VALIDATE limit live in
**`scaffold/PROTO.md`** and ship live via **`skill://elon`** (Plugin B). Read
those for the full protocol — this file does not duplicate it.

## Build, test, validate

```bash
npm ci                            # install deps
npm run typecheck                 # tsc --noEmit (Plugin A TypeScript)
npm test                          # node --test src/*.test.ts
bash scripts/validate-plugins.sh  # structural integrity, both plugins (needs jq)
```

`validate-plugins.sh` checks file presence, manifest shape, and agent/skill
coverage so a broken tree never reaches `omp plugin install`. It needs only bash
+ `jq` (the TypeScript typecheck is a separate step that needs the toolchain).

## CI

`.github/workflows/ci.yml` runs on every push/PR (branches `main`, `dev`) in
three layers, in the `validate` job:

1. `npm run typecheck` (Plugin A TypeScript)
2. `bash scripts/validate-plugins.sh` (structure of both plugins)
3. install both through the real `omp` binary as a smoke test (Plugin B is pure
   markdown — installs without bun; Plugin A is an extension-package whose
   install runs `bun install`)

A second job, `installer-one-liner`, runs the repo's **own** `elon_ko.sh` with
Plugin A pinned to the commit under test — a regression guard for the `#`
github-ref delimiter (a revert to `@` makes the pinned fetch 404). `elon_ko.sh`
bootstraps `omp` + `bun` itself, so that job needs no node/bun setup.

## Releases

`.github/workflows/release.yml` fires on a `v*` tag. After the same typecheck +
validation gate (and an assertion that the tag matches `package.json#version`),
it publishes a GitHub Release with two consumable archives:

| Artifact | Plugin | Local install |
|---|---|---|
| `omp-agent-gate-<ver>.tgz` | A — `npm pack` (honors `package.json#files`) | extract, then `omp plugin install ./package` |
| `orchestrator-agents-<ver>.tar.gz` | B — self-contained marketplace (`.omp-plugin/` + `plugins/`) | extract, then `omp plugin marketplace add ./<dir>` |

omp consumes plugins straight from git, so **the tag is the distribution pin** —
the archives are for offline/pinned/local installs. A `SHA256SUMS` is attached.

```bash
# cut a release (version must match package.json#version)
git tag v1.8.0 && git push origin v1.8.0

# install a pinned version straight from the tag
omp plugin install github:rokicool/omp-agent-template#v1.8.0
```

> **Version sync:** keep `package.json#version`, `.omp-plugin/marketplace.json`
> `metadata.version`, and `plugins[].version` in lockstep before tagging. The
> marketplace version drifted to `1.2.1` in a prior release (fixed in v1.3.1).
> The github-source delimiter is `#` (e.g. `github:owner/repo#v1.8.0`), not
> `@` — `@` separates a marketplace plugin name from its marketplace.

## Pre-releases (testing unreleased work)

`.github/workflows/prerelease.yml` publishes every push to a non-`main` branch
as a GitHub **pre-release** tagged `pr-<branch>-<short-sha>` (idempotent —
re-pushing the same commit updates the same release; a `pr-*` tag never matches
`release.yml`'s `v*` trigger, so it never cuts a production release). Then
`elon_ko.sh <tag>` installs **both** plugins pinned to that exact ref:

- **Plugin A** — `github:<repo>#<tag>`, the same pin mechanism as a stable release.
- **Plugin B** — fetched as the tag's source tarball and registered as a LOCAL
  marketplace under `$OMP_PRERELEASE_DIR` (default `~/.omp-prerelease`), because
  omp marketplaces track the default branch and **cannot** be ref-pinned. The
  directory is **kept** — omp references a local marketplace in place, so
  deleting it would break Plugin B resolution.

```bash
bash elon_ko.sh pr-dev-abc1234
# piped one-liner (-s -- forwards the tag to the downloaded script):
curl -fsSL https://raw.githubusercontent.com/<owner>/omp-agent-template/pr-dev-abc1234/elon_ko.sh | bash -s -- pr-dev-abc1234
```

This is distinct from `OMP_AGENT_REF=<tag>` (Plugin A only; Plugin B still tracks
latest). Re-running `elon_ko.sh` with no argument re-registers the marketplace
from GitHub and returns to the latest stable release. Both modes work on a clean
machine, in a container, and over a prior install — the marketplace is removed
and re-added each run, and Plugin A is uninstalled before the pinned install to
avoid a `DependencyLoop`.

## Contributing

- Keep the two-plugin split: TypeScript/rules → Plugin A (`package.json`);
  agents/skills → Plugin B (`plugins/`).
- Don't weaken a hard layer to fix a prompt-level problem — the gate and the
  agent frontmatter are the contract.
- Update [CHANGELOG.md](./CHANGELOG.md) per release, following the Keep a
  Changelog format.
- Opt-in marker is `.omp/elon.json` (renamed from `.omp/orchestrator.json` in
  v1.3.1 — migrating opted-in projects must rename the file, else the gate
  reverts to dormant).

## Troubleshooting (dev)

- **`DependencyLoop` installing Plugin A.** A different ref of `omp-agent-gate`
  is already locked in `~/.omp/plugins/`. `--force` won't clear it — uninstall
  first: `omp plugin uninstall omp-agent-gate`, then install the new pinned ref.
  `elon_ko.sh` does this automatically on every run.
- **Plugin A install fails `Executable not found in $PATH: "bun"`.**
  Extension-package install runs `bun install`, and the omp binary installer
  ships no bun. Install bun (`curl -fsSL https://bun.sh/install | bash`), or
  install omp with `--source` (brings bun as a side effect).
- **Stale build: Elon tries to `read ./AGENTS.md` / `./PROTO.md` and fails.**
  You're on a stale Plugin A. Reinstall pinned, uninstalling first so the
  changed ref doesn't trip a `DependencyLoop`:
  `omp plugin uninstall omp-agent-gate && omp plugin install github:rokicool/omp-agent-template#v1.8.0`.
  (The framing explicitly tells Elon those files are optional.)
- **`skill://elon` doesn't resolve in a session.** Structural coverage is
  verified by `validate-plugins.sh`; live resolution still depends on omp's
  skills provider scanning the plugin's `skills/` tree. The bundled
  APPEND_SYSTEM framing is sufficient for Elon to route even if it fails.
- **Targeted at** `@oh-my-pi/pi-coding-agent` 16.x (install path re-verified
  against 16.1.16 in a clean container).
- **Plugin code runs in-process, unsandboxed** when loaded. Install only from
  sources you trust. MIT license.
